# TalentScan HR / Employer Agent — QA Protocol v1.1

## 0. Назначение файла

Этот файл задаёт обязательный QA-протокол для всех HR/Employer-карточек TalentScan.

QA проводится перед выдачей пользователю любого HTML-файла, шаблона, роутера, карточки, сравнительной карточки или TeamScan-карточки.

Цель QA: не допустить потери структуры, смешения аудиторий, технической терминологии, подтягивания прошлого контекста, ошибок в мобильной версии, наложения блоков и непрактичных выводов.

---

## 1. Обязательная последовательность QA

Перед финальной выдачей агент обязан пройти 6 уровней проверки:

1. **Source QA** — проверка источников и контекста.
2. **Router QA** — правильно ли выбран тип карточки.
3. **Content QA** — качество HR-содержания.
4. **Client Language QA** — нет ли технического/служебного языка.
5. **Card-Type QA** — проверка по типу карточки.
6. **HTML / Mobile QA** — работоспособность, адаптивность, якоря, отсутствие наложений.

Если ошибка найдена, агент сначала исправляет, затем перепроверяет, и только после этого отдаёт файл.

---

## 2. Source QA — проверка источников и контекста

### 2.1. Новый кейс

Проверить:

- [ ] Это новый кандидат или продолжение прошлого кейса?
- [ ] Не подтянута ли компания из прошлого запроса?
- [ ] Не подтянута ли вакансия из прошлого запроса?
- [ ] Не подтянуты ли выводы по прошлому человеку?
- [ ] Не используется ли прошлый процент соответствия?
- [ ] Если пользователь прислал новую карту, она считается новым человеком, пока не сказано обратное.

### 2.2. Данные компании

Проверить:

- [ ] Есть ли название компании?
- [ ] Есть ли описание продукта / услуги?
- [ ] Есть ли понимание клиента компании?
- [ ] Понятно ли, что именно продаётся / производится / оценивается?
- [ ] Не пересказывается ли компания слишком подробно, если работодатель и так её знает?
- [ ] Контекст компании используется только там, где он влияет на оценку кандидата.

### 2.3. Данные вакансии / роли

Проверить:

- [ ] Есть ли название роли?
- [ ] Есть ли задачи роли?
- [ ] Есть ли KPI или критерии успеха?
- [ ] Есть ли формат работы?
- [ ] Есть ли требования к навыкам?
- [ ] Если KPI/воронка/доля холодных лидов неизвестны, это помечено как “нужно уточнить”.

### 2.4. Данные кандидата

Проверить:

- [ ] Есть имя кандидата?
- [ ] Понятно, на каких данных строится оценка?
- [ ] Не придуманы ли опыт, достижения, резюме или факты биографии?
- [ ] Если данных мало, это честно отражено.

---

## 3. Router QA — проверка типа карточки

Проверить:

- [ ] Один кандидат под одну роль → Candidate for Role.
- [ ] Несколько кандидатов под одну роль → Candidate Comparison.
- [ ] Руководитель + кандидат → TeamScan Manager × Candidate.
- [ ] Сотрудник в текущей роли → Employee Role Fit.
- [ ] Нет кандидата, есть вакансия → Role Criteria Matrix.
- [ ] Нет роли, есть только кандидат → General HR Candidate Card или уточнение.
- [ ] Не смешаны ли карточка кандидата, сравнение и TeamScan в одну нечитаемую страницу.

---

## 4. Content QA — базовое качество HR-содержания

Проверить:

- [ ] В начале есть быстрый HR-вывод.
- [ ] Карточка отвечает на вопрос работодателя: “что делать дальше?”
- [ ] Вывод привязан к конкретной роли, а не к человеку “вообще”.
- [ ] Сильные стороны сформулированы через пользу для роли.
- [ ] Риски сформулированы через рабочие ситуации.
- [ ] Есть вопросы для интервью или 1:1.
- [ ] Есть тестовое задание / практическая проверка, если речь о найме.
- [ ] Есть финальная рекомендация.
- [ ] Есть “что проверить до оффера”.
- [ ] Нет финального приговора без проверки.
- [ ] Проценты выглядят как HR-гипотеза, а не абсолютная истина.
- [ ] Не слишком много пересказа вакансии и компании.
- [ ] Карточка полезна HR/руководителю, а не только аналитически красива.

---

## 5. Client Language QA — клиентский язык

### 5.1. Запрещённые слова в клиентской HTML-карточке

Проверить, что в клиентской части нет:

- [ ] ворота;
- [ ] каналы;
- [ ] сакрал;
- [ ] центр;
- [ ] бодиграф;
- [ ] Манифестирующий Генератор;
- [ ] Проектор;
- [ ] профиль;
- [ ] авторитет;
- [ ] Генный Ключ;
- [ ] соционический тип;
- [ ] ЧС / БЭ / БЛ / ЧИ;
- [ ] электромагнит;
- [ ] доминантный канал;
- [ ] компромиссный канал;
- [ ] служебный блок;
- [ ] эталон агента;
- [ ] правила генерации.

Исключение: эти слова могут быть в закрытых служебных файлах агента, но не в клиентской карточке.

### 5.2. Правильный язык

Проверить, что используются слова:

- [ ] рабочий стиль;
- [ ] сильные стороны;
- [ ] риски;
- [ ] условия раскрытия;
- [ ] стиль управления;
- [ ] обратная связь;
- [ ] адаптация;
- [ ] интервью;
- [ ] тестовое;
- [ ] CRM;
- [ ] follow-up;
- [ ] клиентская коммуникация;
- [ ] результат;
- [ ] следующий шаг.

---

## 6. QA для карточки “Кандидат под конкретную роль”

Проверить:

- [ ] В hero указаны кандидат, компания, роль.
- [ ] Есть быстрый вывод: статус, процент, причина, риск, следующий шаг.
- [ ] Контекст компании и вакансии компактный и не перегружает карточку.
- [ ] Компания и вакансия используются для оценки кандидата, а не пересказываются ради пересказа.
- [ ] Есть карта данных: факты / гипотезы / что уточнить.
- [ ] Есть соответствие задачам роли.
- [ ] Есть сильные стороны кандидата именно под эту роль.
- [ ] Есть риски кандидата именно в этой роли.
- [ ] Есть интервью-вопросы.
- [ ] Есть тестовое задание.
- [ ] Есть условия управления и адаптации.
- [ ] Есть итоговая HR-рекомендация.
- [ ] Нет служебного блока в конце.

Критическая ошибка: карточка выглядит как общий психологический разбор, а не как оценка кандидата под вакансию.

---

## 7. QA для карточки “Сравнение кандидатов на одну роль”

Проверить:

- [ ] В hero указаны кандидаты, компания, роль.
- [ ] Есть быстрый сравнительный вывод.
- [ ] Понятно, кто приоритетный кандидат и почему.
- [ ] Сравнение идёт по одной и той же матрице критериев.
- [ ] Критерии привязаны к вакансии.
- [ ] У каждого кандидата есть сильные стороны.
- [ ] У каждого кандидата есть риски.
- [ ] Есть общие вопросы интервью.
- [ ] Есть разводящие вопросы интервью.
- [ ] Есть единое тестовое задание для честного сравнения.
- [ ] Есть сценарии решения: один кандидат / два кандидата / резерв.
- [ ] Нет сравнения “кто лучше вообще”.
- [ ] Нет служебного блока в конце.

Критическая ошибка: выбор кандидата сделан по общему впечатлению, а не по критериям роли.

---

## 8. QA для TeamScan “Руководитель × кандидат”

Проверить:

- [ ] В hero указаны руководитель, кандидат, компания/роль.
- [ ] Есть быстрый вывод по рабочей связке.
- [ ] Понятно, что это рабочая, а не личная совместимость.
- [ ] Раскрыто, что даёт руководитель.
- [ ] Раскрыто, что даёт кандидат.
- [ ] Есть сильные стороны связки.
- [ ] Есть риски связки.
- [ ] Есть рекомендации по постановке задач.
- [ ] Есть рекомендации по обратной связи.
- [ ] Есть адаптация 7/30/90 или другой прикладной план.
- [ ] Есть вопросы для 1:1 / финального этапа.
- [ ] Есть итоговое HR-решение.
- [ ] Совместимость не используется как самостоятельный отказ или оффер.
- [ ] Нет технических терминов композита.
- [ ] Нет служебного блока в конце.

Критическая ошибка: TeamScan звучит как “они совместимы/несовместимы”, но не даёт руководителю инструкции, как управлять.

---

## 9. HTML / Mobile QA

### 9.1. Структура HTML

Проверить:

- [ ] HTML-файл открывается.
- [ ] Нет битых тегов.
- [ ] Нет пустых секций.
- [ ] Нет случайных служебных комментариев.
- [ ] Нет неактуальных названий прошлых файлов / кандидатов.
- [ ] Название файла соответствует содержанию.

### 9.2. Навигация

Проверить:

- [ ] Все пункты меню ведут к существующим id.
- [ ] Нет пунктов меню на удалённые блоки.
- [ ] Кнопка “Вернуться в начало” работает, если она есть.
- [ ] Sticky / fixed меню не перекрывает заголовки.
- [ ] На мобильном меню не ломает первый экран.

### 9.3. Мобильная адаптация

Проверить:

- [ ] На ширине мобильного экрана блоки не налезают друг на друга.
- [ ] Hero-блок не имеет наложений: проценты, плашки, карточки, текст.
- [ ] Таблицы прокручиваются горизонтально.
- [ ] Карточки становятся в одну колонку.
- [ ] Нет горизонтального скролла всей страницы, кроме таблиц/навигации.
- [ ] Текст читаемый, не слишком мелкий.
- [ ] Верхняя навигация не занимает чрезмерно много места.

### 9.4. Визуальные ошибки, которые обязательно ловить

- [ ] Надпись не налезает на круг/процент.
- [ ] Плашки не перекрывают карточки.
- [ ] Карточки не выходят за контейнер.
- [ ] Секции не накладываются друг на друга.
- [ ] Нет служебного блока в клиентской версии.
- [ ] Нет верхнего бургера, если текущий клиентский эталон требует верхнюю строку без бургера.

---

## 10. Decision QA — качество финального решения

В конце карточки должно быть понятно:

- [ ] вести кандидата дальше или нет;
- [ ] в каком статусе вести: основной кандидат, финалист, резерв, рискованный вариант;
- [ ] что проверить до оффера;
- [ ] какое тестовое дать;
- [ ] какие условия нужны для успеха;
- [ ] какие красные флаги отслеживать;
- [ ] как адаптировать после найма.

Нельзя заканчивать карточку фразой вроде:

> “В целом кандидат интересный”.

Нужно заканчивать конкретным действием:

> “Рекомендация: вести на следующий этап, дать ролевую продажу и CRM-кейс; оффер рассматривать только если кандидат покажет follow-up и управляемость процесса”.

---

## 11. Финальный QA-отчёт в ответе пользователю

После выдачи файла агент должен кратко написать:

1. Что сделано.
2. Что проверено.
3. Что исправлено, если были ошибки.
4. Что осталось спорным / зависит от данных.
5. Ссылку на файл.

Формат короткий, без длинного отчёта.

Пример:

> Готово. Проверил: якоря, мобильную адаптацию, отсутствие служебных блоков, клиентский язык, фокус на HR-решении. Спорное место: оценка зависит от реальных KPI и доли холодных/тёплых лидов.

---

## 12. Fail conditions — когда файл нельзя отдавать

Файл нельзя отдавать пользователю, если:

- [ ] в hero есть визуальное наложение;
- [ ] меню ведёт на несуществующие id;
- [ ] клиентская версия содержит служебный блок;
- [ ] в клиентской версии есть техническая терминология;
- [ ] карточка подтянула прошлый кейс;
- [ ] нет быстрого HR-вывода;
- [ ] нет финального действия для работодателя;
- [ ] сравнение кандидатов сделано по разным критериям;
- [ ] TeamScan используется как основание для отказа без практической проверки;
- [ ] карточка выглядит как описание компании/вакансии, а не оценка кандидата.

Если найден fail condition: исправить, перепроверить, затем выдать.

---

## 13. Короткая версия QA перед каждым файлом

Перед выдачей файла агент должен внутренне пройти мини-чек:

1. Тип карточки выбран правильно?
2. Новый кейс не смешан со старым?
3. Быстрый HR-вывод есть?
4. Компания/вакансия не перегружены?
5. Фокус на кандидате/решении есть?
6. Риски и вопросы есть?
7. Тестовое есть?
8. Клиентский язык без терминов?
9. Служебных блоков нет?
10. Мобильная версия не ломается?
11. Якоря рабочие?
12. Финальное решение практичное?

Если все ответы “да” — файл можно выдавать.
